## 1. Replies were suddenly cut off mid-sentence (用户痛点 #1)

**Root cause.** `server.js` capped every model call at `MAX_TOKENS = 1024`. The live
provider is gpt-5.1 via the UvA proxy — a *reasoning* model whose hidden reasoning
tokens count against the same cap, so even a moderately long visible reply ran out of
budget and was truncated mid-sentence.

**Fix.** `MAX_TOKENS` is now `Number(process.env.LLM_MAX_TOKENS ?? 4096)` (env-
overridable), and the OpenAI-compatible path logs a warning whenever
`finish_reason === 'length'` so a future cap problem is visible in Railway logs
instead of silent. File: `server.js`.

## 2. Replies were too long to read comfortably (用户痛点 #1bis)

**Fix.** The Stage-C **main** prompt's PACING section and the **baseline** prompt now
share a hard, *identical* length rule: at most 2 short paragraphs, ~60–110 words, one
idea/scene per message, at most one question back. The main prompt's COMPONENT 2 also
says a scene should land in 2–3 sentences. Identical wording in both conditions keeps
reply length from becoming a condition confound. Andrea's two Phase-B prompts
(reflective / direct) are **untouched**. File: `lib/prompt.js`.

## 3. Tappable question cards after every reply (用户痛点 #2)

**Fix.** A fixed, neutral pool of 18 questions (`ASK_POOL` in `chat.jsx`) covering
angles a participant may not think of (money, weekends, stress, regrets, first job,
paths not taken, …). After **every** future-self reply, 3 not-yet-used questions are
shown as chips above the composer ("Ideas to ask"); tapping one sends it as the user
turn; the pool recycles when exhausted; free typing always remains. Also added to the
free-continuation chat.

**Design note (deliberate):** the pool is static and identical across conditions —
NOT model-generated — so the nudges cannot differ between main and baseline (no new
confound). Files: `chat.jsx`, `styles.css` (`.ask-row`, `.ask-chip`; the old one-shot
`.suggested` block was removed).

## 4. Restart button covered the bottom-left explanation text (用户痛点 #3)

**Root cause.** The fixed-position `↻ Restart` fab sat exactly on the chat sidebar's
"Why this is a role-play" text.

**Fix.** On desktop chat (≥861px) the fab is hidden; the sidebar's top button (now
labelled "↻ Start over", previously "New session") does the same thing and now asks
for confirmation first (it used to restart instantly on a single click — data-loss
hazard). On mobile, where the sidebar is hidden, the fab remains. Files: `app.jsx`
(`confirmRestart` shared by fab + sidebar), `chat.jsx`, `styles.css`.

## 5. Timer not visible while chatting (用户痛点 #4)

**Fix.** The count-up clock now also renders in the composer footer (next to "Ask
anything…"), i.e. exactly where the participant is looking while typing. The header
clock stays. Still a count-UP clock — Build Plan §16 forbids countdown/pressure cues.
Files: `chat.jsx`, `styles.css`.

## 6. Default reading font → sans (用户痛点 #5)

**Fix.** `COMFORT_DEFAULTS.font` changed `'serif'` → `'sans'` and the localStorage
key bumped `thesis_comfort_v1` → `thesis_comfort_v2` so devices that stored the old
default pick up the new one (other comfort prefs reset once on first visit — accepted
trade-off). Serif remains available in the Aa comfort panel. File: `screens.jsx`.

## 7. Post-test chat looked like a brand-new conversation (用户痛点 #6)

**Root cause.** `FreeContinuation` reused the same server session (model memory was
intact) but rendered an empty thread.

**Fix.** `app.jsx` now passes the Phase-C transcript as a `history` prop;
`FreeContinuation` renders it read-only (slightly dimmed) above a quiet
"Your conversation continues" divider, auto-scrolled to the newest turn. Autosave
still records **only the new free-continuation turns** — the logging separation
(§3.9b: free chat never part of the main analysis) is preserved. Files: `app.jsx`,
`chat.jsx`, `styles.css`.

## 8. Admin: "List failed. — AggregateError" (用户痛点 #7)

**Root cause (live, verified via `/healthz`):** the Railway service variable
`DATABASE_URL` points at **localhost** (it matches the local-dev line in
`.env.example`). Inside the Railway container there is no localhost Postgres, every
connection is refused on all addresses, and Node surfaces that as `AggregateError`
with an empty message. **No code change can fix the live instance — see §9.**

**Code fixes (diagnosability):**
- `lib/db.js`: new `describeDbError()` unwraps AggregateError into the real causes
  (e.g. `connect ECONNREFUSED 127.0.0.1:5432`); new `dbHint()` returns an actionable
  fix line when `DATABASE_URL` is unset or points at localhost. `/healthz` now
  includes both.
- `lib/admin_routes.js`: all list/export/load error responses carry `detail` + `hint`.
- `admin/index.html`: a `DbHealthBanner` probes `/healthz` once and shows a single
  prominent banner ("Database not connected — … Fix: …") across all tabs instead of
  four cryptic per-tab failures; per-request errors also append the hint.
